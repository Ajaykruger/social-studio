## Task 2 — Attach-rendered-reel flow

**Goal:** a reel campaign created in the Create tab is currently blocked
from approval (by design) because its `ugc_video` item has
`reviewAction: "produce_before_review"` and no media. The operator renders
the MP4 with MoneyPrinterTurbo on the studio PC; this task lets them attach
that file so the campaign becomes reviewable.

**Server** — new endpoint in `server/decision-api.mjs`:
`POST /api/campaigns/:campaignId/attach-reel` with JSON body
`{ "filePath": "<absolute or workspace-relative path to the rendered .mp4>" }`.

Behavior (implement in a new `server/attach-reel.mjs` with an injectable
`workspaceRoot`, following the style of `server/generate-campaign.mjs`):

1. Validate campaign id (existing pattern) and that `draft-bundle.json`
   exists and the campaign's review board has a `ugc_video` item.
2. Reject if the campaign already has an `approved-bundle.json` (409).
3. Validate the file: exists, extension `.mp4`, size > 0. Resolve relative
   paths against the workspace root. Reject paths that resolve outside the
   workspace root AND outside `MoneyPrinterTurbo/storage/` — those two
   roots are the only allowed sources.
4. Copy it to `public/social-studio/<campaignId>/review/reel-01.mp4`.
5. Update `review-board/review-board.json`: the `ugc_video` item gets
   `reviewAction: "review_decision_required"` and
   `media.videoUrl: "/social-studio/<campaignId>/review/reel-01.mp4"`.
6. Update `review-packet/review-packet.ui.json`: the pending reel asset gets
   the real `assetUrl` and label `"Reel"`; update `visualReviewSummary`.
7. Update `workflow-status.ui.json`: the "Reel render" stage becomes
   `ready`, blocker text drops the render clause.
8. Append an audit entry (`event: "reel_attached"`, the source path, and
   `allowsSchedulingOrPublishing: false`) to
   `social-studio/audit/<campaignId>.decisions.jsonl`.

**UI** — in `ReviewDecisionScreen.jsx` (or a small new component it
renders): when an asset has `contentType: "ugc_video"` and no `assetUrl`,
show an "Attach rendered reel" box with a text input for the file path and
a button calling the endpoint (add a helper in `src/utils/studioData.js`).
On success call the existing `onDecided`-style reload. Keep it
mobile-friendly.

**Tests** — new `social-studio/tests/attach-reel.test.mjs` using a temp
workspace (copy the fixture style from
`social-studio/tests/generate-campaign.test.mjs`):
- attach succeeds: file copied, board item flipped, packet + status updated,
  audit appended;
- the campaign can then be APPROVED via `runReviewDecisionCycle` (pass
  `workspaceRoot`) with the reel + post evidence gates;
- rejects: missing campaign (404), non-mp4 file, path outside the allowed
  roots, campaign already approved (409), campaign without a ugc_video item.

**Acceptance:** all of the above; full suite green; manual smoke: create a
reel campaign via the API, attach a dummy mp4 from `MoneyPrinterTurbo/storage/`,
see the Review tab unblock.

## Task 4 — Reviewer identity + timestamp hardening

**Goal:** close two known soft spots.

- `social-studio/tools/record-review-decision.mjs`: `approvalTimestamp`
  currently accepts any string of length ≥ 10. Require a valid ISO-8601
  date-time (`Number.isFinite(Date.parse(value))` AND
  `/^\d{4}-\d{2}-\d{2}T/` shape). Keep the error message style.
- `server/decision-api.mjs`: optional reviewer allowlist. If env
  `STUDIO_REVIEWERS` is set (comma-separated, e.g. `Jen,Andre`), reject
  decisions whose `reviewer` is not in the list (case-insensitive match,
  400 with a clear message). If unset, behave as today. Document the
  variable in `README.md` next to the `.env` section.

**Tests:** bad timestamps rejected (tool-level test); allowlist behavior
on/off (API-level test, set the env var around `createDecisionApp` via an
options parameter `reviewers` rather than mutating `process.env` in tests —
read the env in `main()`/`createDecisionApp` default).

**Acceptance:** suite green; approving as "Jen" still works with
`STUDIO_REVIEWERS=Jen,Andre`.

## Task 5 — Deployment package (files only, no actual deployment)

**Goal:** everything needed to put this on a Hetzner CX32 behind Cloudflare
Access, as committed files + a runbook a non-coder can follow. Do NOT
provision anything; produce artifacts.

Create a `deploy/` folder:

- `deploy/DEPLOY.md` — step-by-step runbook: create Hetzner CX32 (Ubuntu
  24.04), create a non-root user, `ufw` allow 22/80/443 only, install Node 22
  + Caddy + git, clone the repo to `/opt/social-studio`, `npm ci`,
  `npm run build`, create `.env` on the server (ANTHROPIC_API_KEY,
  optional STUDIO_REVIEWERS), install the systemd unit, configure Caddy,
  put Cloudflare Access (Google login for two emails) in front, nightly
  backup cron. Write for a non-technical operator: exact commands, in
  order, with what success looks like after each step.
- `deploy/social-studio.service` — systemd unit running
  `node server/decision-api.mjs` as the app user with
  `Environment=HOST=127.0.0.1 PORT=4810`, `Restart=on-failure`,
  `WorkingDirectory=/opt/social-studio`.
- `deploy/Caddyfile` — reverse proxy `studio.<domain>` → `127.0.0.1:4810`
  with automatic HTTPS; placeholder domain clearly marked.
- `deploy/backup.sh` — tar `social-studio/generated`,
  `social-studio/audit`, `social-studio/handoff`, and `public/social-studio`
  to `/var/backups/social-studio/<date>.tar.gz`, keep 14 days. Plus the
  cron line in DEPLOY.md.
- IMPORTANT: the server must keep binding 127.0.0.1; Caddy is the only
  public listener, and Cloudflare Access fronts Caddy. State in DEPLOY.md
  that the app must never be exposed without the access layer.

**Acceptance:** files present and internally consistent (paths, user names,
ports match across runbook/unit/Caddyfile); `bash -n deploy/backup.sh`
passes; no real hostnames or secrets committed.
